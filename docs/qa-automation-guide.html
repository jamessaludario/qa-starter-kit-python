1About Playwright

First: what is QA automation?

When testers check a website by hand — clicking buttons, filling forms, and reading the screen to confirm everything works — that is manual testing. QA automation means writing small programs (called test scripts) that do that clicking, typing, and checking for you. The computer repeats the same checks perfectly, every time, in minutes instead of hours.

What is Playwright?

Playwright is a free, open-source tool made by Microsoft that lets your code control a real web browser: open pages, click, type, and read what appears on screen. You write the instructions in Python (Playwright also supports JavaScript, Java, and C#, but this guide uses Python).

Why do people choose Playwright?

  • It controls all major browsers — Chromium (the engine behind Chrome and Edge), Firefox, and WebKit (the engine behind Safari) — with the same code.
  • Auto-waiting. Web pages take time to load. Older tools failed if a button wasn't ready yet, forcing testers to add "sleep 5 seconds" everywhere. Playwright automatically waits for elements to be ready before acting on them, which makes tests far more reliable.
  • Fast and modern. It handles today's dynamic, JavaScript-heavy websites well.
  • Great tooling. It can record your clicks and generate code for you (codegen), record a full "trace" of a test run you can replay later, and take screenshots and videos automatically.
  • Headless or headed. Tests can run with a visible browser window ("headed", great for watching and debugging) or invisibly in the background ("headless", great for speed and for servers with no screen).

How Playwright compares to Selenium

You may have heard of Selenium, the older and most widely known browser automation tool. Both are good; Playwright is generally easier for beginners because auto-waiting is built in, setup is simpler (browsers are installed with one command), and its error messages are clearer. Selenium's advantages are its longer history and larger ecosystem. For a new project in 2026, Playwright is a very safe choice.

The pieces you will use

ToolWhat it does
playwrightThe library that drives the browser (click, type, read the page).
pytestThe test runner: it finds your test files, runs them, and reports pass/fail. It is the most popular test framework in Python.
pytest-playwrightA plugin that connects the two: it hands each test a ready-to-use browser page and adds handy command-line options like --headed.

2Prerequisites

Things to install first

  1. Python 3.9 or newer. Download from python.org/downloads.
    Windows users — important! On the first screen of the Python installer, tick the checkbox "Add python.exe to PATH". If you skip this, the python command won't work in your terminal.
    Verify it installed by opening a terminal and running: 
    python --version
    You should see something like Python 3.12.x.
  2. pip — Python's package installer. It comes bundled with Python. Verify with: 
    pip --version
  3. A code editor. Visual Studio Code (free) is the usual choice. Install its Python extension from the Extensions panel.
  4. A terminal. Windows: PowerShell (built in). macOS/Linux: Terminal. You will type commands here; every command in this guide runs in a terminal.

Knowledge that helps (but is not required)

  • Basic Python — variables, functions, if statements, imports. If these are new, a few hours on any free Python tutorial is enough to start.
  • Basic HTML — knowing that a page is made of elements like <button>, <input>, and <a> (links) makes locators (Section 6) much easier to understand.
  • Browser DevTools — right-click any element on a web page and choose Inspect to see its HTML. You will use this constantly to find elements.

3Installation and setup

Step 1 — Create a project folder

mkdir my-first-automation
cd my-first-automation

Step 2 — Create a virtual environment

A virtual environment (venv) is a private copy of Python just for this project, so its packages don't clash with other projects on your machine. Always use one. 

# Create it (makes a folder called .venv)
python -m venv .venv

# Activate it - Windows (PowerShell):
.venv\Scripts\Activate.ps1

# Activate it - macOS / Linux:
source .venv/bin/activate

When it's active, your terminal prompt shows (.venv) at the start. You must activate it every time you open a new terminal for this project.

PowerShell says "running scripts is disabled"? Run this once, then try activating again: 
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

Step 3 — Install the packages

Create a file named requirements.txt listing what the project needs:

requirements.txt 
playwright
pytest
pytest-playwright
pytest-html==3.2.0
pytest-rerunfailures

Then install everything in one go:

pip install -r requirements.txt
PackageWhy you need it
playwrightDrives the browser.
pytestFinds and runs your tests.
pytest-playwrightGives each test a ready browser page + CLI options.
pytest-html==3.2.0Writes a shareable HTML report of each run. (Version 3.x is pinned because 4+ builds its report with JavaScript, which shows up empty in viewers that block scripts.)
pytest-rerunfailuresOptional: retries a failed test automatically — useful for slow or flaky sites.

Step 4 — Install the browsers

Playwright downloads its own copies of the browsers (it does not use your everyday Chrome). This is a one-time step: 

playwright install

To save disk space you can install only Chromium: playwright install chromium

Step 5 — Recommended project structure

You could put everything in one file, but real projects separate concerns into layers. This structure keeps tests readable and easy to maintain: 

my-first-automation/
|-- .venv/               (virtual environment - never edit, never commit to git)
|-- pytest.ini           (test runner configuration - Section 4)
|-- conftest.py          (shared fixtures - Section 8)
|-- constants.py         (URLs, timeouts, other fixed values)
|-- requirements.txt     (package list)
|-- pages/               (page objects: ALL element locators live here)
|   |-- base_page.py
|   |-- login_page.py
|-- helpers/             (reusable multi-step journeys, e.g. "log in as user X")
|-- utils/               (test data generators, small utilities)
|-- tests/               (the actual tests - assertions live here)
|   |-- test_login.py
|-- reports/             (generated HTML reports land here)
The layering rule (worth memorizing) Tests contain assertions ("is the welcome message visible?"). Helpers contain journeys ("register a new user"). Pages contain locators ("the login button is get_by_role('button', name='Login')"). Never put locators directly in tests — when the website changes, you fix one page file instead of fifty tests. This pattern is called the Page Object Model (POM).

4Configurations

pytest.ini — the control panel

pytest.ini sits in your project root and tells pytest how to behave every time you run it, so you don't retype the same options. A practical example:

pytest.ini 
[pytest]
; Which folder(s) contain the tests
testpaths = tests

; Options applied automatically on every run:
;   --browser chromium        -> run in Chromium
;   --html=...                -> write an HTML report
;   --self-contained-html     -> put CSS inside the one report file (easy to share)
;   --reruns 2                -> retry a failed test up to 2 more times
;   --reruns-delay 2          -> wait 2 seconds between retries
addopts = --browser chromium --html=reports/report.html --self-contained-html --reruns 2 --reruns-delay 2

; Named categories for tests (see Section 9)
markers =
    smoke: quick checks of the most critical paths
    regression: the full, thorough suite

Useful command-line options from pytest-playwright

These can go in addopts (permanent) or be typed per-run (temporary):

OptionEffect
--browser chromiumChoose the browser: chromium, firefox, or webkit. Repeat the flag to run in several.
--headedShow the browser window while tests run (default is invisible/headless).
--slowmo 500Pause 500 ms between each action so you can watch what happens.
--screenshot only-on-failureSave a screenshot whenever a test fails.
--video retain-on-failureRecord video, keep it only for failures.
--tracing retain-on-failureRecord a Playwright trace (replayable recording) for failures.
--base-url https://mysite.comSet a base URL so tests can use page.goto("/login").

conftest.py — shared setup code

conftest.py is a special file pytest loads automatically. It holds fixtures (reusable setup/teardown blocks — explained fully in Section 8) and project-wide hooks. Anything defined here is available to every test without imports.

constants.py — one home for fixed values

constants.py 
# Every URL and timeout lives here, so changing an environment
# (e.g. staging -> production) is a one-line edit.
BASE_URL = "https://automationexercise.com"
DEFAULT_TIMEOUT = 10_000   # milliseconds - Playwright times are in ms

5Creating test scripts

The naming rules (pytest finds tests by name)

  • Test files must start with test_ — e.g. test_login.py
  • Test functions inside them must also start with test_ — e.g. def test_login_with_valid_password():

Anything that doesn't follow this pattern is simply ignored by pytest.

Your first test

tests/test_first.py 
# "page" is provided automatically by pytest-playwright:
# a fresh, ready-to-use browser tab for this one test.
# "expect" is Playwright's assertion helper (Section 7).
from playwright.sync_api import Page, expect


def test_homepage_has_correct_title(page: Page):
    # 1. ARRANGE / ACT - go to the website
    page.goto("https://playwright.dev")

    # 2. ASSERT - check the page title contains "Playwright".
    # If it does, the test PASSES. If not, it FAILS with a clear message.
    expect(page).to_have_title("Playwright")  # substring/regex also possible

Run it:

pytest tests/test_first.py --headed

A browser window opens, visits the site, checks the title, and closes. Congratulations — that's an automated test.

The anatomy of a good test: Arrange, Act, Assert

Almost every test follows the same three beats (the "AAA pattern"):

  1. Arrange — get everything ready (open the page, prepare test data).
  2. Act — do the one thing you're testing (click "Login", submit the form).
  3. Assert — check the outcome ("Logged in as Maria" is visible).

A realistic example using the Page Object Model

First, a page object — the only place locators are allowed to live:

pages/login_page.py 
from playwright.sync_api import Page


class LoginPage:
    """Knows WHERE things are on the login page and HOW to use them.
    Tests never touch locators directly - they call these methods."""

    def __init__(self, page: Page):
        self.page = page
        # All locators defined once, in one place:
        self.email_input = page.get_by_placeholder("Email Address")
        self.password_input = page.get_by_placeholder("Password")
        self.login_button = page.get_by_role("button", name="Login")
        self.error_message = page.get_by_text("Your email or password is incorrect!")

    def open(self):
        self.page.goto("https://automationexercise.com/login")

    def login(self, email: str, password: str):
        self.email_input.fill(email)
        self.password_input.fill(password)
        self.login_button.click()

Then the test — short, readable, and free of locators:

tests/test_login.py 
from playwright.sync_api import Page, expect
from pages.login_page import LoginPage


def test_login_with_wrong_password_shows_error(page: Page):
    # ARRANGE
    login_page = LoginPage(page)
    login_page.open()

    # ACT
    login_page.login("someone@example.com", "definitely-wrong-password")

    # ASSERT
    expect(login_page.error_message).to_be_visible()

Rules of thumb for writing tests

  • One behavior per test. "Login with wrong password shows an error" is one test; don't also verify the signup form in the same function.
  • Descriptive names. test_login_with_wrong_password_shows_error tells you what broke without opening the file. test_1 tells you nothing.
  • Tests must be independent. Each test creates whatever data it needs and cleans up after itself, so tests can run in any order — and one failure doesn't cascade.
  • Never use time.sleep(). Use expect(...) assertions, which wait intelligently (Section 7 explains why).

6How to access elements (locators)

What is a locator?

A web page is built from HTML elements — buttons, text boxes, links, images. Before your script can click a button, it must first find that button among the hundreds of elements on the page. A locator is the "address" you give Playwright so it can find the right element.

Think of it like giving directions: "the button labeled Login" or "the text box with placeholder Email Address". In code: 

# Create a locator (nothing happens on the page yet - it's just the address)
login_button = page.get_by_role("button", name="Login")

# Use it - NOW Playwright finds the element and clicks it
login_button.click()
Locators are "lazy" and self-updating A locator doesn't search the page when you create it — only when you act on it (.click(), .fill(), or inside expect()). Each action re-finds the element, so locators keep working even on pages that redraw themselves.

The recommended locators (use these first)

Playwright's get_by_* methods find elements the way a human would — by their visible label, role, or placeholder. They survive code refactors much better than locators tied to HTML internals.

LocatorFinds elements by...Example
get_by_role() Their role (button, link, heading, checkbox...) and visible name. The best default choice. page.get_by_role("button", name="Sign up")
get_by_text() The text they display. page.get_by_text("Welcome back")
get_by_label() The form label next to an input. page.get_by_label("Password")
get_by_placeholder() The gray hint text inside an empty input. page.get_by_placeholder("Email Address")
get_by_title() Their title attribute (tooltip text). page.get_by_title("Close dialog")
get_by_alt_text() An image's alternative text. page.get_by_alt_text("Company logo")
get_by_test_id() A data-testid attribute developers add specifically for tests. Very stable — ask your developers to add them! page.get_by_test_id("submit-order")

CSS and XPath locators (when the above aren't enough)

page.locator() accepts CSS selectors (the same syntax used to style pages) and XPath. You'll need these for elements without good labels: 

# CSS: by id (the # symbol) - ids should be unique on a page
page.locator("#susbcribe_email")

# CSS: by class (the . symbol)
page.locator(".shopping-cart")

# CSS: by tag + attribute
page.locator("input[name='search_product']")

# CSS: element inside another element (a space means "descendant")
page.locator("#cart_items .product-name")

# XPath (starts with // ) - powerful but harder to read; prefer CSS
page.locator("//button[text()='Checkout']")

How to discover an element's address

  1. Open the page in your browser.
  2. Right-click the element and choose Inspect. DevTools opens with its HTML highlighted, e.g. <button class="btn btn-primary" data-qa="login-button">Login</button>
  3. Pick the most stable, human-readable address: here, role button with name Login, or the data-qa attribute via page.locator("[data-qa='login-button']").

Or let Playwright suggest one for you — its recorder shows a recommended locator for anything you hover:

playwright codegen https://automationexercise.com

Narrowing down: chaining and filtering

When several elements match, get more specific:

# Search INSIDE another element (chaining)
page.locator(".navbar").get_by_role("link", name="Products")

# Keep only matches that contain certain text (filtering)
page.get_by_role("listitem").filter(has_text="Blue Top")

# Pick by position - use as a last resort, order can change
page.get_by_role("link", name="View Product").first
page.get_by_role("row").nth(2)          # third match (counting starts at 0)

Common actions on a located element

ActionWhat it does
.click()Click it.
.fill("text")Clear a text box and type the text.
.press("Enter")Press a keyboard key while focused on it.
.check() / .uncheck()Tick / untick a checkbox.
.select_option("USA")Choose from a dropdown (<select>).
.hover()Move the mouse over it (to open hover menus).
.inner_text()Read the text it displays.
.scroll_into_view_if_needed()Scroll until it is on screen.
Priority order when choosing a locator 1) get_by_role  2) other get_by_* methods  3) get_by_test_id  4) simple CSS  5) XPath (last resort). The higher on this list, the less likely your test breaks when the page's internals change.

7Assertions

What is an assertion?

An assertion is the checking step — the moment your test says "I expect X to be true; if it isn't, fail this test." Without assertions a script just clicks around; assertions are what make it a test.

Use expect() — it waits for you

Playwright's expect() assertions are web-first: they don't check once and give up. They keep re-checking (by default for up to 5 seconds) until the condition becomes true or time runs out. This absorbs the little delays that are normal on real websites — a success message that takes 800 ms to appear simply passes. 

from playwright.sync_api import expect

# Keeps checking for up to 5s until the element is visible - then passes.
expect(page.get_by_text("Order placed!")).to_be_visible()

# Need longer for a slow operation? Override per-assertion:
expect(page.get_by_text("Report ready")).to_be_visible(timeout=15_000)
Never use time.sleep() to wait A fixed sleep is always wrong twice: too short on a slow day (test fails randomly) and too long on a fast day (test wastes time). expect() waits exactly as long as needed. If you're tempted to sleep, there is always an expect() that says what you're really waiting for.

The assertions you'll use most

AssertionChecks that...
expect(locator).to_be_visible()the element is shown on screen.
expect(locator).to_have_text("Done")its text is exactly "Done".
expect(locator).to_contain_text("Don")its text contains "Don".
expect(locator).to_have_value("42")an input box contains "42".
expect(locator).to_be_enabled()it can be clicked (not grayed out).
expect(locator).to_be_checked()a checkbox is ticked.
expect(locator).to_have_count(3)exactly 3 elements match the locator.
expect(page).to_have_title("Home")the browser tab title is "Home".
expect(page).to_have_url("https://site.com/cart")the current address matches.

Negating: checking something is NOT there

# After removing the item, the cart should show the empty message
# and the product row should be gone:
expect(page.get_by_text("Cart is empty!")).to_be_visible()
expect(page.locator("#product-1")).not_to_be_visible()

What a failure looks like

When an assertion fails, pytest prints exactly what was expected vs. what was found:

AssertionError: Locator expected to be visible
Actual value: hidden
Call log:
  - expect(locator).to_be_visible with timeout 5000ms
  - waiting for get_by_text("Order placed!")

Read these carefully — they usually tell you precisely what went wrong and where.

8Setup and teardown

The concepts

  • Setup — work done before a test runs: open a browser, log in, create a test user, load data.
  • Teardown — cleanup done after a test finishes (pass or fail): delete the test user, log out, close the browser.

Good teardown is what keeps tests independent: every test starts from a clean slate no matter what the previous test did.

pytest fixtures — how setup/teardown is written

In pytest, setup and teardown live in fixtures: functions marked with @pytest.fixture. A test "requests" a fixture simply by naming it as a parameter. Everything before the yield line is setup; everything after it is teardown.

conftest.py 
import pytest
from playwright.sync_api import Page


@pytest.fixture
def logged_in_page(page: Page):
    # ---------- SETUP (runs before the test) ----------
    page.goto("https://automationexercise.com/login")
    page.get_by_placeholder("Email Address").fill("tester@example.com")
    page.get_by_placeholder("Password").fill("secret123")
    page.get_by_role("button", name="Login").click()

    # yield hands control to the test, passing it the ready page
    yield page

    # ---------- TEARDOWN (runs after the test, even if it FAILED) ----------
    page.get_by_role("link", name="Logout").click()
tests/test_account.py 
from playwright.sync_api import expect


# Just naming "logged_in_page" as a parameter triggers the whole fixture.
def test_dashboard_greets_the_user(logged_in_page):
    expect(logged_in_page.get_by_text("Logged in as")).to_be_visible()
You've already been using a fixture! The page parameter in every earlier example is a fixture, provided by pytest-playwright. Its setup launches a fresh browser tab; its teardown closes it. That's why you never wrote "open browser" or "close browser" yourself.

Fixture scope — how often setup runs

ScopeSetup/teardown runs...Typical use
function (default)once per testfresh page, fresh test data
classonce per test classshared state for a group of related tests
moduleonce per fileexpensive setup shared by one file's tests
sessiononce per entire runstart a mock server, load config
@pytest.fixture(scope="session")
def api_token():
    # Fetched once, reused by every test in the run.
    return fetch_token_from_api()
Where fixtures live Put fixtures used by many files in conftest.py — pytest auto-discovers them, no import needed. Fixtures used by a single file can live at the top of that file.

9Categorization of tests

Why categorize?

Once you have dozens of tests, you rarely want to run all of them every time. After a small change you might run just the quick "smoke" checks (2 minutes) and save the full "regression" suite (30+ minutes) for later. Categories make that possible.

Markers — pytest's labels

A marker is a label you stick on a test with a decorator:

import pytest


@pytest.mark.smoke
def test_homepage_loads(page):
    ...

@pytest.mark.regression
def test_full_checkout_journey_with_new_address(page):
    ...

# A test can carry several markers:
@pytest.mark.smoke
@pytest.mark.cart
def test_add_item_to_cart(page):
    ...

Register your markers (avoid typo bugs)

Declare every marker in pytest.ini. Unregistered markers produce warnings, and — worse — a typo like @pytest.mark.smoek would otherwise silently create a new category and your test would stop running with the smoke suite.

pytest.ini (excerpt) 
[pytest]
markers =
    smoke: quick checks of the most critical user paths
    regression: the full, thorough suite
    cart: anything involving the shopping cart
addopts = --strict-markers   ; turn unknown markers into ERRORS - highly recommended

Running by category

# Only smoke tests
pytest -m smoke

# Everything EXCEPT slow regression tests
pytest -m "not regression"

# Smoke tests that also involve the cart
pytest -m "smoke and cart"

# Either category
pytest -m "smoke or cart"

Common suite names in QA teams

SuitePurposeWhen it runs
Smoke"Is the app fundamentally alive?" 5-15 fast tests of the most critical paths (login, homepage, checkout).On every code change / deployment.
Regression"Did anything anywhere break?" The full suite.Nightly, or before a release.
Feature areas (cart, auth, search...)Everything about one part of the app.When that area changes.

Other useful built-in markers

# Skip a test entirely (always give a reason!)
@pytest.mark.skip(reason="Feature removed in v2, test pending rewrite")
def test_old_wishlist(page): ...

# Skip only under some condition
@pytest.mark.skipif(sys.platform == "win32", reason="Safari not available on Windows")
def test_webkit_specific(page): ...

# "We KNOW this fails - there's an open bug." Reported as xfail, not failure.
@pytest.mark.xfail(reason="BUG-1234: totals rounding error")
def test_cart_total_rounding(page): ...

10How to run tests and check reports

Running tests — the commands you'll actually use

# Run EVERYTHING pytest can find
pytest

# Run one file
pytest tests/test_login.py

# Run one specific test inside a file
pytest tests/test_login.py::test_login_with_wrong_password_shows_error

# Run tests whose NAME contains a keyword
pytest -k "cart"

# Run a category (Section 9)
pytest -m smoke

# Watch the browser while it runs, slowed down
pytest --headed --slowmo 500

# More detail in the terminal (-v = verbose); -s shows print() output
pytest -v -s

# Stop at the first failure (handy while debugging)
pytest -x

# Re-run only the tests that failed last time
pytest --lf

Reading the terminal output

tests/test_login.py ..F.                                          [100%]

=================== short test summary info ====================
FAILED tests/test_login.py::test_login_shows_username - AssertionError: ...
=============== 1 failed, 3 passed in 42.10s ====================
  • Each . is a passing test; F is a failure; s means skipped; x means expected failure (xfail); R means the test was retried.
  • Above the summary, every failure gets a full breakdown: the exact line that failed, what was expected, and what was actually found.

The HTML report (pytest-html)

If your pytest.ini includes --html=reports/report.html --self-contained-html (Section 4), every run writes a report automatically. Otherwise add those flags to any run by hand.

  1. Run your tests as usual: pytest
  2. Open reports/report.html in any browser (double-click it in your file explorer).

The report shows:

  • A summary bar — how many passed, failed, skipped, or were rerun, and total duration.
  • A results table — one row per test with its outcome and duration.
  • Failure details — the full error text for every failed test, and (if you enabled --screenshot only-on-failure) a screenshot of the page at the moment of failure.

Because of --self-contained-html, the report is a single file — you can attach it to an email or a ticket and it will display anywhere.

Allure — a richer, dashboard-style report (optional)

Allure produces an interactive report with graphs, history, and per-step timelines. Teams often adopt it once suites grow. Two-part setup: the pytest plugin (pip install allure-pytest) records results, and the Allure command-line tool renders them: 

# 1. Record results while testing
pytest --alluredir=reports/allure-results

# 2. Turn the results into a report and open it
allure generate reports/allure-results -o reports/allure-report --clean
allure open reports/allure-report
Note: the allure command itself is a separate install (it needs Java, or on Windows: scoop install allure). Start with pytest-html; add Allure when you feel the need for prettier analytics.

What to do when a test fails

  1. Read the error message. It names the assertion, the locator, and what was found instead.
  2. Look at the failure screenshot (if enabled) — often the page shows an unexpected popup or error banner that explains everything.
  3. Re-run just that test with a visible browser: pytest tests/test_x.py::test_name --headed --slowmo 500 and watch what happens.
  4. Decide: bug or broken test? A failing test means either the app broke (report a bug — automation just did its job!) or the app changed legitimately and the test needs updating.

11Additional info

Debugging tools built into Playwright

  • Codegen (test recorder). Opens a browser; as you click around it writes Python code for every action. Great for learning locator syntax: 
    playwright codegen https://automationexercise.com
  • Playwright Inspector (step-through debugger). Runs your test one action at a time with a live locator explorer: 
    # Windows (PowerShell)
$env:PWDEBUG = "1"; pytest tests/test_login.py

# macOS / Linux
PWDEBUG=1 pytest tests/test_login.py
  • Trace Viewer (flight recorder). Run with --tracing retain-on-failure and each failure saves a trace.zip. Open it to scrub through every step with before/after screenshots, console logs, and network calls: 
    playwright show-trace path/to/trace.zip
    This is the single most powerful way to diagnose a failure that only happens on the CI server.

Dealing with flaky tests

A flaky test passes sometimes and fails sometimes without any code change — usually caused by timing (the site was slow) or shared state (two tests fighting over the same account). Defenses, in order of preference:

  1. Replace any fixed waits with expect() assertions (Section 7).
  2. Make every test create its own data — e.g. register a fresh user with a unique, timestamped email instead of sharing one account.
  3. As a safety net, auto-retry failures with pytest-rerunfailures: --reruns 2 --reruns-delay 2. Retried tests are flagged in reports, so you can still spot and fix the flakiness rather than hide it.

Generating unique test data

utils/data.py 
import uuid


def unique_email() -> str:
    """A fresh email every call, e.g. qa-3f2a1c@example.com,
    so tests never collide over the same account."""
    return f"qa-{uuid.uuid4().hex[:6]}@example.com"

Good habits checklist

  • Keep all locators in page objects — never in tests.
  • Wait with expect(), never time.sleep().
  • One behavior per test; name tests after that behavior.
  • Tests create their own data and clean up after themselves.
  • Register markers and use --strict-markers.
  • Add .venv/, reports/, and __pycache__/ to .gitignore.
  • Start small: automate 5 solid smoke tests before attempting 50.

Running tests automatically (CI) — a peek ahead

Once your suite is stable, teams wire it into Continuous Integration (CI) — a service like GitHub Actions or Jenkins that runs the tests automatically on every code change and posts the report. Playwright runs headless on CI servers out of the box, so the suite you built locally is already CI-ready. That's a topic for its own guide, but know that it's the natural next step.
